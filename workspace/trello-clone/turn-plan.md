## Learnings (built reality vs. requirements — this is where this turn's assessment differs from turn 9's)

1. **FR8, as currently written, contradicts the feature it describes.** `01-understand/requirements.md` FR8 reads: *"Fields autosave on change/blur (no separate 'Save' button); closing the modal never discards edits already made."* But the reviewed, APPROVED `task-007` implementation does the opposite (`04-build/reviews/task-007-review.md` item 5, `events.js:32-58,193-208`): there **is** an explicit Save button, and Cancel/X/overlay-close all **discard** unsaved edits. `backlog.md:41` had punted the save/close decision to the developer with only a comment requirement; the developer made a reasonable, documented choice, and code-reviewer approved it — but nobody circled back to correct the FR8 text itself. Nine turns and two more reviewed tasks (task-011, this new assessment) have now passed with the requirements doc simply describing behavior the app does not have. This is a genuine spec-vs-reality drift, not a cosmetic nit, and belongs in `requirement-refiner`'s in-scope work this turn: reconcile FR8's text to the shipped Save/Cancel modal (recommended — it's simpler, conventional, and already live) or file a bug to rebuild autosave.
2. **NFR7 (visible, non-blocking warning to the user on persistence failure) is unimplemented.** `04-build/src/public/js/storage.js:52` catches save failures with `console.warn` only — no UI-visible warning ever reaches the user, and there is no evidence any task or review checked this path. This NFR has existed unimplemented since task-002 (turn 1) and has never been named in any prior turn's gap-analyst output until now — a genuine miss carried silently for 9 turns.
3. **`architecture.md`'s stale "no dependencies" line is now an 8-turn-old process smell.** It's cheap (one line) and keeps getting logged as a footnote instead of being given to an agent as an actual action. Recommend the next `tech-writer` or `architect` touch fix it directly rather than flag it a ninth time.
4. **The 9-task, zero-bounce review streak plus a never-run `ui-tester`, taken together, is the real risk here.** Every code review across 10 turns has been APPROVED first-pass; the one time dynamic verification ran (turn 7 Playwright) it also passed, but covers only the boards-overview surface (~1/9 of built tasks). The FR8 drift above is a concrete instance of something that passed static review cleanly (explicitly called "defensible" by the reviewer) while nonetheless leaving the requirements doc simply wrong about the product's behavior — exactly the class of discrepancy that a real user-facing exploratory pass (`ui-tester`) or a broader regression suite (`test-writer`) is positioned to catch and static review is not.
